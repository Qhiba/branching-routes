# ran_0202_risks.md — Risk Register

**Feature:** Variants_on_nodes_and_Options_on_choices
**Generated:** 2026-04-18

## RISK-VNO-01 — Referential integrity gap in `deleteFlag` / `deleteStatus`

**Description:** The new `variants[].requires`, `options[].requires`, `options[].flags_set`, and `options[].status_set` fields create new locations that reference flag and status IDs. The existing `deleteFlag` / `deleteStatus` scans check only `edges[].condition.conditions` and `node.data.flags_set` / `node.data.status_set`. If the new fields are not added to the scan, a flag or status can be silently deleted while still referenced inside a variant or option — breaking any future simulation that uses them.

**Likelihood:** High — The gap is structural; every new flag or status deletion after Phase 1 will miss the new fields until the scan is patched.

**Impact:** High — Silent orphaned references in `requires` conditions will evaluate incorrectly during simulation. No warning will surface to the designer.

**Early detection signal:** After Phase 1, open the app, add a variant with a `requires` flag clause, then delete that flag from the Flags tab. If no blocking warning is shown, RISK-VNO-01 has materialised.

**Mitigation:** Extending `deleteFlag` and `deleteStatus` is a hard requirement of Phase 1. Acceptance criteria for Phase 1 include explicit verification of the blocking dialog for flag/status used in variant or option `requires`.

## RISK-VNO-02 — `ChoiceNode` crash on legacy nodes with no `data.options`

**Description:** All existing choice nodes in saved files have `node.data` without an `options` field. If `ChoiceNode.jsx` reads `data.options.map(...)` or `data.options.length` without a null guard, every legacy choice node in the canvas will throw a runtime error on render, crashing the canvas.

**Likelihood:** High — Every existing save file will trigger this if the guard is missing.

**Impact:** High — Canvas becomes non-functional for any project created before this feature.

**Early detection signal:** Load an existing project (or create a choice node and reload before Phase 1 is complete). If the canvas shows a blank or React error boundary, the guard is missing.

**Mitigation:** Every consumer of `data.options` must use `data.options ?? []`. This is enforced as a Phase 2 acceptance criterion. A `loadGraph` migration stamp (setting `options: []` if absent) is a secondary mitigation — decision made in Phase 1.

## RISK-VNO-03 — React Flow handle ID collision breaks connect events

**Description:** Each per-option source handle on `ChoiceNode` is registered with a `handleId` equal to the option's `id` (e.g. `opt-{uuid}`). React Flow uses the `handleId` to populate `connection.sourceHandle` in the `onConnect` callback. If two handles on the same node share a `handleId`, or if a handle ID matches a React Flow internal key, connect events will misroute or be suppressed.

**Likelihood:** Low — IDs are generated by `generateId('opt')` with a UUID; collisions within a single node are near-impossible. React Flow internal keys use different naming conventions.

**Impact:** High — A misrouted connect event creates an edge with a wrong `optionId`, linking the wrong option to an edge silently.

**Early detection signal:** On a choice node with two options, draw an edge from option 1's handle. Inspect the edge in `EdgeInspector`. If the reported option is option 2 or no option, a handle ID collision or misroute has occurred.

**Mitigation:** Use the `opt-` prefix for all option IDs (distinct from `n-`, `e-`, `f-`, `sp-`, `p-`, `c-`). Validate in Phase 2 by creating two options and confirming distinct handles. Document the prefix convention.

## RISK-VNO-04 — `deleteOption` leaves dangling `optionId` references on edges

**Description:** When a designer deletes an option from a choice node, any edges that have `optionId` pointing to that option's ID become orphaned. The deleted option no longer exists, but the edge still carries its ID. `EdgeInspector` will attempt to look up the label and find nothing, rendering a broken or empty "Connected from option" field.

**Likelihood:** Medium — Designers will routinely add, rearrange, and delete options during authoring.

**Impact:** Medium — Does not break graph structure or simulation routing (edge still connects source/target), but the inspector shows incorrect metadata.

**Early detection signal:** Create a choice node with one option. Connect an edge from that option's handle. Then delete the option. Select the edge — if `EdgeInspector` shows a broken label or crashes, RISK-VNO-04 has materialised.

**Mitigation:** `deleteOption` in `narrativeStore` must also remove all edges where `edge.optionId === optionId` within the same `set()` call. This is a Phase 1 requirement already listed in the file map.

## RISK-VNO-05 — `ChoiceNode` re-render storms from `data.options` subscription

**Description:** `ChoiceNode` currently uses targeted selectors from `simulationStore` and a single edge-count selector from `narrativeStore`. Adding a subscription to `data.options` (which is an array object) risks breaking `React.memo` if the selector returns a new array reference on every store update, causing all choice nodes to re-render on every unrelated mutation (e.g., editing a flag name).

**Likelihood:** Medium — Array selectors that reconstruct objects on every call are a common React performance pitfall in Zustand.

**Impact:** Medium — At small graph sizes, imperceptible. At 30+ choice nodes, each unrelated edit triggers dozens of redundant renders, contradicting RISK-01 mitigation strategy.

**Early detection signal:** In Phase 2, use React DevTools Profiler. Edit a flag name while a canvas with 5+ choice nodes is open. If all choice nodes re-render, the selector is not stable.

**Mitigation:** `ChoiceNode` must read `data.options` from the `data` prop already passed by React Flow (which is derived from the store and compared by React Flow's own memoisation), not from a new `useNarrativeStore` selector. If a direct store selector is needed, use `shallow` equality from Zustand or a stable derived selector.

## RULE CANDIDATE — RC-01: Sub-array CRUD via `updateNode` patch pattern

This feature introduces a pattern of managing nested sub-arrays inside `node.data` via dedicated actions (`addVariant`, `updateVariant`, `deleteVariant`, `addOption`, `updateOption`, `deleteOption`) rather than exposing the arrays directly. This is distinct from the existing flat node property updates done via `updateNode`.

This pattern should be considered for formalisation as an architecture rule once it has been validated through implementation. **Do not add to `architecture_rules.md` yet — defer to 0208 Document.**
