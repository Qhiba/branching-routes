# Campaign_Sheets — Data Model Impact

---

## Does this feature add new fields to any entity?

Yes — but only to the campaign layer, not to the narrative layer.

### New entity: Campaign

A campaign object is a first-class entity in the new `campaignStore`. It has no equivalent in the current data model.

```
Campaign {
  id:              string           // "camp-{uuid}" — generated by generateId('camp')
  name:            string           // user-supplied, e.g. "good_ending_run"
  createdAt:       string           // ISO timestamp
  updatedAt:       string           // ISO timestamp
  campaignSchemaVersion: 1          // versioning field for future-proofing
  snapshot: {
    activeNodeId:        string | null
    seenNodeIds:         string[]
    traversedEdgeIds:    string[]
    flagOverrides:       { [flagId]: boolean }
    statusOverrides:     { [statusId]: number }
  }
}
```

### New `campaignStore` state shape

```
{
  campaigns: { [campaignId]: Campaign }   // the campaign dictionary
  activeCampaignId: string | null         // which campaign is loaded
}
```

### Changes to `narrativeStore` entities

**NONE.** The schema for `common`, `choice`, `ending`, `edges`, `flag`, `status`, `path`, `chapter`, and `meta` is completely unchanged. `schemaVersion` remains `4`.

### Changes to `simulationStore` state

Two small additions to support campaign-awareness:

| Field | Change | Detail |
|-------|--------|--------|
| `activeCampaignId` | ADD | String or null — mirrors `campaignStore.activeCampaignId`, held here so campaign-active components don't need to cross-import the store. Alternatively, components read it directly from `campaignStore`. **Decision for Phase 2:** prefer reading from `campaignStore` to avoid duplication. |

The existing state fields (`isCampaignActive`, `activeNodeId`, `seenNodeIds`, `traversedEdgeIds`, `currentFlagValues`, `reachableEdgeIds`, `reachableNodeIds`, `nodeStates`, `selectedOptionId`, `sandboxOverrides`, `orphanedNodeIds`, `unreachableNodeIds`) are **unchanged in shape**.

---

## Is every addition strictly additive?

Yes. No existing entity gains a new required field. No existing field is removed or renamed. The narrative `schemaVersion: 4` is not bumped.

The only structural additions are:
1. The new `campaignStore` (an entirely new Zustand store — greenfield).
2. An `activeCampaignId` parameter to `enterCampaign()` (optional; existing call sites with zero arguments continue to work, seeding from `narrativeStore` defaults as before).

---

## Does the export/import round-trip survive this change?

### JSON export (no campaigns)
Identical to current behavior. `exportProject(graphData)` produces a `schemaVersion: 4` JSON file. No change.

### ZIP export (campaigns present)
A new bundle format is introduced. It is a `.zip` file containing:
```
project_title.zip
├── datamodel.json     (schemaVersion: 4 — unchanged JSON payload)
└── campaigns/
    ├── good_ending_run.json    (campaignSchemaVersion: 1)
    └── chapter_2_test.json     (campaignSchemaVersion: 1)
```

`datamodel.json` inside the ZIP passes through the same `importProject` migration and sanitization chain. Campaign files are validated independently against `campaignSchemaVersion: 1`.

A project exported as ZIP can be re-imported and fully restored. A project exported as JSON continues to import identically — no campaigns are loaded.

### Legacy file compatibility
Completely preserved. v1–v4 `.json` files import through the unchanged migration chain. They produce a campaign-less project, which is correct behavior.

---

## Which entity IDs or prefixes does this touch?

| Prefix | Entity | New? |
|--------|--------|------|
| `camp-` | Campaign | **NEW** — introduced by this feature |
| `n-`, `e-`, `f-`, `s-`, `p-`, `c-`, `opt-`, `cond-` | All existing narrative entities | Unchanged |

The `camp-` prefix is the only new ID prefix. It is distinct from all existing prefixes and will not collide with React Flow internal keys.
