**Area**: Zustand stores (`graphStore.js`, `simulationStore.js`), ID generation (`uuid.js`), theme tokens (`tokens.css`)

**Date**: 12-04-2026

## 0. Migration Declaration

**Does this refactor touch any persisted key, field name, or export format?**

**YES** — the following persisted keys are potentially in scope:

| Persisted Key | Current Location | Risk |
|---|---|---|
| `schemaVersion` | `graphStore.exportGraph()` L178 | Top-level export key |
| `meta` (object + `createdAt`, `updatedAt`, `title`) | `graphStore.exportGraph()` L179–183 | Nested export key |
| `nodes` | `graphStore.exportGraph()` L184 | Top-level export key |
| `edges` | `graphStore.exportGraph()` L185 | Top-level export key |
| `flags` | `graphStore.exportGraph()` L186 | Top-level export key |
| `id` (on every node, edge, flag) | Generated by `uuid.js`, stored inside arrays | Value format: UUID v4 string |

**Migration sub-phase is MANDATORY for any of the above keys if they are renamed, restructured, or their format changes, before Self-Review proceeds.**

Undeclared keys (e.g., `selectedNodeId`, `selectedEdgeId`, `snapToGrid`) are memory-only and remain under full hard stop protection for behavioral correctness but not persistence.

## 1. Behavioral Invariants

### BI-01 — Adding a node assigns `isStartNode: true` only when the node array is empty
- **Why**: The first node auto-becomes the start node. All other nodes default to `isStartNode: false`. If this breaks, simulations cannot start.
- **Test**: Add a node to an empty graph → `isStartNode === true`. Add a second → `isStartNode === false`.

### BI-02 — `setStartNode(id)` sets exactly one start node
- **Why**: Simulation expects `nodes.find(n => n.data.isStartNode)` to return exactly one node.
- **Test**: Call `setStartNode(id)` → only that node has `isStartNode: true`, all others `false`.

### BI-03 — `deleteNode(id)` cascades to remove all connected edges
- **Why**: Leaving orphan edges creates broken simulation paths and export corruption.
- **Test**: Delete a node → no edge references it as `sourceId` or `targetId`.

### BI-04 — `deleteNode(id)` clears `selectedNodeId` if it matches the deleted node
- **Why**: UI would point to a ghost node, causing null-reference errors in inspectors.
- **Test**: Select a node, delete it → `selectedNodeId === null`.

### BI-05 — `deleteEdge(id)` clears `selectedEdgeId` if it matches
- **Why**: Same as BI-04 for edges.
- **Test**: Select an edge, delete it → `selectedEdgeId === null`.

### BI-06 — `addEdge` rejects edges from ending nodes (AR-12)
- **Why**: Ending nodes are terminal. Allowing outgoing edges breaks simulation invariant.
- **Test**: Attempt `addEdge(endingNodeId, anyId)` → throws Error.

### BI-07 — `addEdge` rejects duplicate edges between same source/target
- **Why**: Duplicate edges create ambiguous simulation choices.
- **Test**: Add same edge twice → second call throws Error.

### BI-08 — `deleteFlag` blocks deletion when the flag is referenced in edges or nodes
- **Why**: Deleting a referenced flag leaves dangling `flagId` pointers in conditions and side effects.
- **Test**: Create flag, reference it in an edge condition → `deleteFlag` returns `{ blocked: true, references: [...] }`.

### BI-09 — `addFlag` validates name against `/^[a-zA-Z0-9_]+$/` (AR-02)
- **Why**: Prevents injection risks and ensures safe condition evaluation.
- **Test**: `addFlag('bad name!', ...)` → throws Error.

### BI-10 — Simulation side effect order: edge first, then destination node (AR-11)
- **Why**: Deterministic flag mutation order. Reversing it changes game logic outcomes.
- **Test**: Create edge with `sideEffects: [set flag X = 1]` and destination node with `sideEffects: [add flag X + 5]`. Advance → flag X === 6 (not 5 then set to 1).

### BI-11 — Simulation `start()` initializes flags from `graphStore.flags[].defaultValue`
- **Why**: Simulation must begin from authored defaults, not stale values.
- **Test**: Set flag default to `42`, start simulation → `currentFlagValues[flagId] === 42`.

### BI-12 — Simulation `reset()` clears all ephemeral state to initial values
- **Why**: AR-08 requires clean reset. Stale simulation state pollutes the next run.
- **Test**: Run simulation, reset → all fields match initial state shape.

### BI-13 — Simulation never mutates `graphStore` (AR-08)
- **Why**: Simulation is read-only over the authored graph.
- **Test**: Run full simulation → snapshot `graphStore` before and after → deep equal.

### BI-14 — `exportGraph` produces `schemaVersion: 1` at top level (AR-09)
- **Why**: `importProject()` in `fileSystem.js` L68 hard-checks `data.schemaVersion !== 1` and throws on mismatch.
- **Test**: Export → output contains `schemaVersion: 1`.

### BI-15 — `exportGraph` formats `createdAt` and `updatedAt` as DD-MM-YYYY strings
- **Why**: This is the persisted format. Changing it would make old exports incompatible with any future reader that expects this format.
- **Test**: Export → `meta.createdAt` matches `/^\d{2}-\d{2}-\d{4}$/`.

### BI-16 — `loadGraph` resets selection state to null
- **Why**: Loading a new graph with stale selection IDs causes ghost-selection bugs.
- **Test**: Select a node, load a different graph → `selectedNodeId === null`.

### BI-17 — `evaluateCondition(null, flags)` returns `true` (unconditional edge)
- **Why**: Edges with no condition must always be traversable.
- **Test**: `evaluateCondition(null, {}) === true`.

### BI-18 — `evaluateCondition` defaults to AND when `operator` is not `'OR'`
- **Why**: Implicit AND is the current contract. Changing default logic changes game behavior silently.
- **Test**: Condition with two clauses, no operator → both must pass.

## 2. Data Contract Invariants

### DC-01 — Export JSON top-level shape
```json
{ "schemaVersion": 1, "meta": {...}, "nodes": [...], "edges": [...], "flags": [...] }
```
- **Consumed by**: `fileSystem.js:importProject()`, `graphStore.loadGraph()`
- **Breaks if changed**: Import rejects the file (schemaVersion check), or loadGraph gets empty arrays.

### DC-02 — Node shape
```json
{ "id": "uuid", "type": "common|ending", "position": { "x": N, "y": N }, "data": { "label": "", "content": "", "isStartNode": bool, "sideEffects": [] } }
```
- **Consumed by**: `simulationStore` (reads `data.isStartNode`, `data.sideEffects`), `GraphCanvas.jsx`, `NodeInspector.jsx`, `StoryNode.jsx`
- **Breaks if changed**: Simulation can't find start node, side effects don't fire, components crash on missing fields.

### DC-03 — Edge shape
```json
{ "id": "uuid", "sourceId": "uuid", "targetId": "uuid", "label": "", "condition": null | { "operator": "AND|OR", "clauses": [{ "flagId": "uuid", "comparator": "==|!=|>|>=|<|<=", "value": any }] }, "sideEffects": [{ "flagId": "uuid", "operation": "set|add|subtract", "value": any }] }
```
- **Consumed by**: `simulationStore` (evaluates conditions, applies side effects), `EdgeInspector.jsx`, `ConditionalEdge.jsx`, `conditionEvaluator.js`
- **Breaks if changed**: Condition evaluation fails, side effects silently stop working, edges render incorrectly.

### DC-04 — Flag shape
```json
{ "id": "uuid", "name": "string", "type": "string", "defaultValue": any }
```
- **Consumed by**: `simulationStore.start()` (reads `defaultValue`), `FlagManager.jsx`, `deleteFlag` referential check
- **Breaks if changed**: Simulation initializes with wrong defaults, flag management UI breaks.

### DC-05 — ID format: UUID v4 string
- **Consumed by**: Every cross-reference (edge→node via `sourceId`/`targetId`, condition→flag via `flagId`, sideEffect→flag via `flagId`)
- **Breaks if changed**: All referential integrity collapses. Old saved files become unloadable if ID format diverges without migration.

### DC-06 — Condition clause references flags by `flagId` (not flag name)
- **Consumed by**: `conditionEvaluator.js`, `simulationStore` flag state keyed by flag ID
- **Breaks if changed**: Condition evaluation would fail to match any flag.

### DC-07 — CSS variable naming convention in `tokens.css`
- **Consumed by**: `global.css`, `App.css`, all component stylesheets
- **Breaks if changed**: Styles break silently — no runtime error, just invisible CSS failures.

## 3. Load-Bearing Assumptions Inventory

### LBA-01 — Synchronous cross-store read via `getState()`
`simulationStore` calls `useGraphStore.getState()` synchronously inside `start()` and `advance()`. This is load-bearing because the simulation assumes it gets a consistent snapshot of the graph at the moment of each call.
- **Status**: MUST REMAIN TRUE. If graph mutations become async or batched, simulation could read stale/partial state.

### LBA-02 — Flat ID strings with no entity-type prefix
The code generates raw UUID v4 without prefixes. `ran_0401_understand.md` notes that `example_datamodel.json` shows prefixed IDs (`n001`, `e001`, `f001`), but the actual code does not. This divergence is acknowledged but the system works with flat IDs.
- **Status**: MUST REMAIN TRUE unless migration sub-phase handles both formats. If prefixes are added, all `find(n => n.id === id)` lookups still work (string comparison), but **import of old files with unprefixed IDs must still function**.

### LBA-03 — `isStartNode` is stored on every node's `data` object
`simulationStore.start()` L46 does `nodes.find(n => n.data && n.data.isStartNode)`. This assumes exactly one node in the array has `isStartNode: true`.
- **Status**: MUST REMAIN TRUE. Moving `isStartNode` off the node data object requires updating simulation and all components that check it.

### LBA-04 — `deleteFlag` traverses edge conditions AND node/edge side effects
The referential integrity check at `graphStore.js:110-129` assumes the specific nested structure `e.condition.clauses[].flagId` and `e.sideEffects[].flagId` and `n.data.sideEffects[].flagId`.
- **Status**: MUST REMAIN TRUE. If these structures change shape, `deleteFlag` silently stops detecting references and allows dangling pointers.

### LBA-05 — UI state (`selectedNodeId`, `selectedEdgeId`) co-located with graph state
`deleteNode` and `deleteEdge` clear selection in the same `set()` call. This is atomic because they share a store.
- **Status**: LOAD-BEARING for atomicity. If UI state moves to a separate store (as suggested in the refactor), the cross-store selection-clearing must be explicitly coordinated. This is called out as **Hidden Complexity** in `ran_0401_understand.md §6`.

## 4. Acceptable Change Surface

The following can be safely restructured:

| Change | Why It's Safe |
|---|---|
| **Extract UI state (`selectedNodeId`, `selectedEdgeId`, `snapToGrid`) into a dedicated `uiStore`** | These fields are memory-only, never persisted. However, cross-store coordination for selection clearing (LBA-05) must be implemented. |
| **Rename internal helper functions** inside stores (e.g., `computeReachable`, `applySideEffects`) | These are module-private, not exported or consumed externally. |
| **Restructure `tokens.css` variable values** (change `#0f1117` to a different hex) | Values can change freely; only variable **names** are load-bearing (DC-07). |
| **Add new CSS variables** to `tokens.css` | Additive-only change, no existing consumer is affected. |
| **Refactor `formatTs` helper** in `exportGraph` | Internal implementation detail, as long as output format remains DD-MM-YYYY (BI-15). |
| **Split store files** into action-grouping modules that are composed back | As long as the public API surface (`useGraphStore`, `useSimulationStore`, their state shape and methods) remains identical. |
| **Move `window.useGraphStore` debug exposure** or remove it | Dev-only convenience, not part of the application contract. |
| **Refactor `fileSystem.js` internals** | As long as it still calls `JSON.stringify` on the exact export shape and validates `schemaVersion` on import. |

## 5. Hard Stop Conditions

The refactor must be **immediately halted** if any of the following occur:

| # | Condition | Why |
|---|---|---|
| HS-01 | **Export JSON shape changes** (keys renamed, nested differently, keys removed) | Breaks all existing saved files. `fileSystem.js` import will reject or silently lose data. |
| HS-02 | **`schemaVersion` is removed, changed, or no longer validated on import** | Violates AR-09. Import becomes unsafe. |
| HS-03 | **Node/edge/flag internal shape changes** without updating ALL consumers | Causes runtime crashes across components and simulation. |
| HS-04 | **ID format changes** without backward-compatible import handling | Existing saved files become unloadable. |
| HS-05 | **Condition format changes** (`clauses`, `operator`, `comparator`, `flagId`) | `conditionEvaluator.js` and simulation break silently. |
| HS-06 | **Side effect execution order changes** (must be: edge first, then destination node) | Violates AR-11. Changes game logic outcomes. |
| HS-07 | **`simulationStore` begins mutating `graphStore` state** | Violates AR-08. Authored graph data gets corrupted by simulation. |
| HS-08 | **Circular import introduced between store files** | Violates AR-06. Causes Zustand initialization failure at runtime. |
| HS-09 | **CSS variable names in `tokens.css` change** without updating all consumers | Silently breaks styling across the entire application. |

**Exemptions per §0**: Keys listed in the Migration Declaration (`schemaVersion`, `meta`, `nodes`, `edges`, `flags`, entity `id` values) are exempt from HS-01 and HS-04 **only if** a migration sub-phase is completed before Self-Review. All other hard stops apply unconditionally.

## 6. Pre-Refactor Verdict

### **PROCEED WITH CAUTION**

**Rationale**: The refactor area is viable and the change surface is well-defined, but the following specific risks require active monitoring:

1. **UI State Extraction (LBA-05)**: Moving `selectedNodeId`/`selectedEdgeId` to a `uiStore` breaks the current atomic selection-clearing in `deleteNode`/`deleteEdge`. The cross-store coordination replacement must be implemented and tested before any other work proceeds. This is the highest-risk item in the refactor.

2. **Persistence Boundary Proximity**: The stores contain both persisted data (the export shape) and ephemeral state. Any structural change to the store risks accidentally touching the export path. The refactor must verify export output after every structural change.

3. **Deep Referential Integrity in `deleteFlag`**: The flag deletion logic at `graphStore.js:110-129` traverses deeply nested structures. If the refactor changes how conditions or side effects are stored (even internally), this traversal must be updated in lockstep.

4. **`simulationStore` ↔ `graphStore` coupling**: The simulation reads graph state synchronously. If the refactor introduces any indirection (e.g., selector layer, derived store), it must preserve synchronous `getState()` semantics.

**The contract is clear. The refactor may proceed with the above risks actively monitored.**
