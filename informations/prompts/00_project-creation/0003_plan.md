## ROLE
You are a senior software architect designing the initial architecture for a new project. 
There is no existing codebase. You are building the foundation that all future work will stand on.

## CONTEXT
Read the scope document first:
`ran_0002_scope.md`

## TASK
Write an implementation plan with these sections:

### 1. Architecture Rules
Propose the initial architecture rules for this project. These are the
rules all future code must follow. Cover:
- Naming conventions (files, variables, entities)
- State management approach
- Data format and structure rules
- Import/export constraints
- Any framework-specific patterns to enforce

Aim for 5–10 rules. Each rule must be one sentence, testable, and specific.

### 2. Initial Data Model
Define the data structures this project will use:
- Entity types and their fields
- Relationships between entities
- The export/save format (JSON structure)
- What a minimal valid data file looks like (example JSON)

### 3. File Map
For each file being **created**:
- File W
- Purpose — what it does and why it exists
- Key exports or interfaces it exposes
- Dependencies — what it imports from other files in this project

### 4. Phases
Ordered list of build steps. Each phase must:
- Have a clear name
- State the **Goal** (why this phase exists, in one sentence)
- State what it **Produces** (files, components, data structures)
- State the **Acceptance Criteria** (Done when: ...)
- State what the next phase depends on from 
- Verification — what the user does to confirm this phase worked, in plain language. Verification must match what was actually built:
  - Foundation phase → "run the app and confirm no errors"
  - Feature phase → "open the app and [specific action the user takes]"
  - Data phase → "check that [specific record or file] exists"
  - Never write a verification step the user cannot perform themselves.
- Each phase will be executed independently and in order — one phase per Execute run
- Declare which reference files this phase needs (architecture, datamodel, filemap)

### 5. Risk Register
3–5 things that could go wrong and how to catch them early.
Expand on the assumptions from Scope Q8 — do not just repeat them.
