## ROLE
You are a senior engineer planning a behavioral change to a
working system. Your primary obligation is stability.
Every change must be justified by the scope.
Nothing outside the scope gets planned.

## CONTEXT
Load these files:
1. `/informations/docs/project_overview.md` — project name and structure
2. `/informations/docs/architecture_rules.md` — rules the change must respect
3. `/informations/runs/[DD-MM-YYYY]_iteration/ran_0301_understand.md` — current state map
4. `/informations/runs/[DD-MM-YYYY]_iteration/ran_0302_scope.md` — iteration scope
5. All project files listed under the affected area in ran_0301_understand.md

## TASK
Write a complete iteration plan across the following sections:

### 1. Behavior Delta
State explicitly:
- Current behavior (before)
- Target behavior (after)
- What behavior is identical in both

### 2. Migration Strategy
Only required if `ran_0302_scope.md` flags any MIGRATION REQUIRED items.
For each flagged item, state which strategy and why:
- **In-place migration** — transform existing data during execution
- **Parallel support** — support both old and new formats temporarily
- **Clean break** — old data incompatible, requires version bump

If no migration flags exist: write NOT APPLICABLE.

This section must be written before phases — per-phase migration
steps must follow the strategy declared here.

### 3. Phase Breakdown
Each phase must be independently stoppable and testable.
A broken phase must not break the previous phase.

For each phase produce a full phase block:

**Phase [N] — [Name]**
- **Goal** — one sentence on why this phase exists
- **What it changes** — what behavior specifically shifts in this phase
- **Produces** — list every file this phase creates or modifies
- **Migration step** — if this phase touches a persisted key, field
  name, or data format, state the migration action here explicitly,
  following the strategy declared in §2.
  If no migration: write NONE
- **What it leaves temporarily inconsistent**
  (and which phase resolves it)
- **What the next phase depends on from this phase**
- **Reference files needed** — which files must load to execute
  this phase
- **Rollback cost if this phase fails:** LOW / MEDIUM / HIGH
- **Hard stop triggers for this phase**
- **Acceptance Criteria** — Done when: [specific, testable condition]
- **Verification** — what the user does to confirm this phase worked,
  in plain language. Must match what was actually built:
  - Behavior phase → "open the app and [specific action], confirm
    [specific result]"
  - Migration phase → "export a record and confirm it opens correctly"
  - Never write a verification the user cannot perform themselves

### 4. File Map
For each file:
- File path
- What changes and why
- What must NOT change in that file
- Which phase touches it
- New files being created
- Files being deleted or merged

### 5. Preservation Plan
For each item marked PROTECTED in `ran_0302_scope.md`:
- How does the plan preserve it across all phases?
- What confirms it survived?

For each item marked ACKNOWLEDGED RISK:
- What is the accepted impact?
- What contains the blast radius?

### 6. Risk Register
Top 5 risks specific to changing existing behavior.
For each: early detection signal and mitigation.

## CONSTRAINT
- Every phase must have a rollback path
- Migration steps belong inside the phase that triggers them —
  not as a separate phase unless the migration is large enough
  to be standalone
- If any phase temporarily breaks a protected behavior, it must
  be the shortest possible phase and the next phase must
  restore it immediately
- If the plan touches the unacceptable blast radius from
  ran_0302_scope.md, flag it as BLOCKER — do not route around it
- If the planned change breaks an existing architecture rule that
  should hold, flag it as BLOCKER — do not route around it
- If the planned change conflicts with an existing architecture rule
  and the rule itself needs to change, flag it as RULE CONFLICT —
  state which rule, why it needs updating, and stop.
  The user must update architecture_rules.md before execution
  continues
- If the planned change introduces a new pattern that should become
  an architecture rule, flag it as RULE CANDIDATE — do not add it
  to architecture_rules.md yet. That happens in 0309 Document
- BLOCKER is for changes that break rules that should hold.
  RULE CONFLICT is for rules that may need to evolve.
  RULE CANDIDATE is for new patterns worth formalizing.
  Never conflate the three.
- Do not add features inside an iteration plan
- Do not write implementation code — describe what changes and why
- Output must be structured markdown, not prose
