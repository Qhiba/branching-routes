## TASK
Write a complete refactor plan:

### 1. Structural Delta
State explicitly:
- Current structure (before)
- Target structure (after)
- What is identical in both

### 2. Phase Breakdown
Each phase must be independently shippable.
A broken phase must not break the previous phase.
For each phase:
- Phase name
- Files modified in this phase, explicit list of file paths
- What it restructures
- What it leaves temporarily inconsistent 
  (and when that gets resolved)
- Rollback cost if this phase fails: LOW / MEDIUM / HIGH
- Hard stop triggers for this phase

### 3. Migration Strategy
How does existing data survive the restructure?
Options:
- In-place migration (transform existing data)
- Parallel support (support both old and new temporarily)
- Clean break (old data incompatible — requires version bump)

State which strategy and why.

### 4. File Map
For each file:
- File path
- What structurally changes
- What must not change in that file
- New files being created
- Files being deleted or merged

### 5. Invariant Preservation Plan
For each behavioral invariant from Audit First:
- How does this phase plan preserve it?
- What test confirms it?

### 6. Risk Register
Top 5 risks specific to structural refactoring.
For each: early detection signal and mitigation.

### 7. Save Task Completion Report
Save your report inside `/informations/runs/[DD-MM-YYYY]_refactor/ran_0404_plan.md`

## CONSTRAINT
- Every phase must have a rollback path
- If any phase unavoidably breaks a behavioral invariant temporarily, it must be the shortest possible phase and the next phase must restore it immediately
- If the plan requires breaking a hard stop condition, flag it as BLOCKER — do not route around it
- Do not add features inside a refactor plan
