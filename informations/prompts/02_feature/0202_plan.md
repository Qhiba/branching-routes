## ROLE
You are a senior software architect planning a new feature.
You protect the existing system while extending it cleanly.
Every addition must be justified by the scope.
Nothing outside the scope gets planned.

## CONTEXT
Load these files:
1. `/informations/docs/project_overview.md` — project name and structure
2. `/informations/docs/architecture_rules.md` — rules the feature must respect
3. `/informations/runs/[DD-MM-YYYY]_feature/ran_0201_scope.md` — feature scope
4. All project files listed under "Files to touch" in ran_0201_scope.md

## TASK
Write a complete feature plan across the following sections:

### 1. Feature Delta
State explicitly:
- What the system does not have now
- What it will have after this feature
- What existing behavior is identical in both

### 2. Data Model Impact
- Does this feature add new fields to any entity?
- Is every addition strictly additive?
- Does the export/import round-trip survive this change?
- Which entity IDs or prefixes does this touch?

If no data model changes: write NOT APPLICABLE.

### 3. Phase Breakdown
Each phase must be independently stoppable and testable.
A broken phase must not break the previous phase.

For each phase produce a full phase block:

**Phase [N] — [Name]**
- **Goal** — one sentence on why this phase exists
- **What it adds** — what specifically is introduced in
  this phase
- **Produces** — list every file this phase creates or modifies
- **What it leaves temporarily incomplete**
  (and which phase completes it)
- **What the next phase depends on from this phase**
- **Reference files needed** — which files must load to
  execute this phase
- **Rollback cost if this phase fails:** LOW / MEDIUM / HIGH
- **Hard stop triggers for this phase**
- **Acceptance Criteria** — Done when: [specific, testable
  condition]
- **Verification** — what the user does to confirm this phase
  worked, in plain language. Must match what was actually built:
  - Logic phase → "open the app and [specific action], confirm
    [specific result]"
  - UI phase → "open the app and confirm [specific element]
    appears"
  - Never write a verification the user cannot perform themselves

### 4. File Map
For each file:
- File path
- What changes and why
- What must NOT change in that file
- Which phase touches it
- Whether it is new or existing

### 5. Integration Points
For every existing component or feature this touches:
- What it currently does
- How the new feature connects to it
- What must not change in how it works

### 6. Risk Register
Top 5 risks specific to introducing this feature.
For each: early detection signal and mitigation.

## Save Report
Save each section as a separate file:

All files save to: `/informations/runs/[DD-MM-YYYY]_feature/`

## CONSTRAINT
- Every phase must have a rollback path
- If the feature requires breaking an architecture rule,
  flag it as BLOCKER — do not route around it
- If a phase is HIGH complexity, suggest breaking it into
  sub-phases
- Do not add scope beyond what ran_0201_scope.md defines
- Do not write implementation code — describe what changes
  and why
- Output must be structured markdown
I forgot to check this on @contextScopeItemMention, check for this condition
- If the feature conflicts with an existing architecture rule
  and the rule itself needs to change, flag it as
  RULE CONFLICT — state which rule, why it needs updating,
  and stop. The user must update architecture_rules.md
  before execution continues
- If the feature introduces a new pattern that should become
  a rule, flag it as RULE CANDIDATE — do not add it to
  architecture_rules.md yet. That happens in 0208 Document
- BLOCKER is for features that break rules that should hold.
  RULE CONFLICT is for rules that may need to evolve.
  RULE CANDIDATE is for new patterns worth formalizing.
  Never conflate the three.
